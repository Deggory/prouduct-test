## 1. What Is This Repo? (Explained Like You're 10)

Imagine a robot car. It has cameras. Every time it takes a photo (20 times per second), a computer looks at the photo and thinks: "Where is the road? Where are other cars? How should I turn the wheel?" That thinking is called **AI inference** — running an AI model to get an answer.

This repo is the software that runs on a special computer chip called **RK3588** that sits inside a device called **ExoPilot**. The ExoPilot is a small box you stick behind your car's rearview mirror. It plugs into the car and helps it drive itself.

**Why is this different from a normal computer?**

A normal computer's brain (CPU) is like a person who can do many things but not super fast. The RK3588 chip has a special extra brain called an **NPU** (Neural Processing Unit) that is ONLY good at one thing — running AI models — but it does it **10 times faster** than the CPU alone.

This repo makes full use of that NPU. That's the whole point.

**What makes this NOT a normal openpilot?**

- Vanilla openpilot runs AI models on a Snapdragon chip (in the comma 3 device)
- The sunnypilot-pc repo (Orange Pi 5) runs AI on a GPU using a software called tinygrad
- **This repo** runs AI on the Rockchip NPU using RKNN format models — totally different approach

### Trick 1 — NPU Core Lock (Dedicated Core, Zero Contention)

**File:** `selfdrive/modeld/runners/rknn_platform.py`

The driving_vision model is **pinned exclusively to Core 0** of the 3-core NPU. No other model is allowed on Core 0 while driving. This means it never waits for another model to finish.

```python
# selfdrive/modeld/runners/rknn_platform.py

class NPUMask(IntEnum):
    AUTO     = 0   # RKNN_NPU_CORE_AUTO  (shares with everything — SLOW)
    CORE_0   = 1   # RKNN_NPU_CORE_0     (exclusive — FAST)
    CORE_1   = 2   # RKNN_NPU_CORE_1
    CORE_2   = 4   # RKNN_NPU_CORE_2
    CORE_0_1 = 3   # Both core 0+1 combined (for huge models)

# RK3588 NPU allocation — who goes on which core:
NPU_ALLOCATION_MAP = {
    PlatformType.RK3588: {
        # Core 0: driving_vision ONLY (exclusive! never shared)
        'driving_vision': NPUMask.CORE_0.value,   # ← 10ms because it owns Core 0

        # Core 1: all stereo/perception models
        'stereod':   NPUMask.CORE_1.value,
        'yolo':      NPUMask.CORE_1.value,
        'ppliteseg': NPUMask.CORE_1.value,
        'scene3d':   NPUMask.CORE_1.value,

        # Core 2: policy + mono detection
        'policy':    NPUMask.CORE_2.value,   # ← 5ms here
        'autospeed': NPUMask.CORE_2.value,
        'monod':     NPUMask.CORE_2.value,
    }
}
```

This core mask gets passed to `rknn.init_runtime(core_mask=npu_core)` when the model is first loaded.

**Orange Pi 5 equivalent:**

### Trick 2 — GPU Does Preprocessing WHILE NPU Runs (Pipelining)

**File:** `selfdrive/modeld/models/commonmodel_pyx.pyx`

The Mali GPU handles YUV decode + affine warp (the image preparation). This uses the GPU OpenCL in parallel while the NPU is running the previous frame. Key: GPU and NPU are **separate chips** — they run truly in parallel.

```cython
# selfdrive/modeld/models/commonmodel_pyx.pyx  (Cython, compiled to .so)

cdef class DrivingModelFrame(ModelFrame):
    """Wraps C++ DrivingModelFrame — runs OpenCL on Mali GPU."""

    def prepare(self, VisionBuf buf, float[:] projection):
        """
        GPU step: NV12 (camera format) → planar uint8 (model format)
                  + affine warp (rotation/perspective correction)
        This runs on Mali OpenCL asynchronously.
        Time: ~5ms on Mali-G610
        """
        cdef mat3 cprojection
        memcpy(cprojection.v, &projection[0], 9 * sizeof(float))
        cdef cl_mem* data
        data = self.frame.prepare(
            buf.buf.buf_cl,   # ← DMA-BUF from camera (zero-copy, no memcpy!)
            buf.width, buf.height, buf.stride, buf.uv_offset,
            cprojection       # ← 3×3 warp matrix from calibration
        )
        return CLMem.create(data)   # ← Returns GPU buffer (still on GPU)

    def buffer_from_cl(self, CLMem in_frames):
        """
        Copy GPU result → CPU numpy array (for NPU input).
        ~1ms copy, unavoidable (RKNN needs CPU memory).
        """
        cdef unsigned char* data2
        data2 = self.frame.buffer_from_cl(in_frames.mem, self.buf_size)
        return np.asarray(<cnp.uint8_t[:self.buf_size]> data2)
```

**In modeld.py, the sequence is:**
```python
# modeld.py — inside model.run()

# Step A: GPU warp (async, ~5ms on Mali-G610)
imgs_cl = {
    name: self.frames[name].prepare(bufs[name], transforms[name].flatten())
    for name in self.vision_input_names
}

# Step B: Copy GPU result → CPU numpy (blocks here ~1ms)
vision_inputs = {}
for name in self.vision_input_names:
    frame_input = self.frames[name].buffer_from_cl(imgs_cl[name])
    vision_inputs[name] = frame_input.reshape((1, 12, 128, 256)).astype(np.uint8)

# Step C: NPU inference (blocks here ~10ms — the dedicated Core 0)
t0 = time.perf_counter()
vision_result = self.npu.infer('driving_vision', vision_inputs)
vision_time = (time.perf_counter() - t0) * 1000
# vision_time prints: "Inference times: vision=9.8ms, policy=4.7ms"
```

**Orange Pi 5 equivalent (you already have CLContext + DrivingModelFrame):**
```python
# Same .pyx file works! Just add RKNN instead of tinygrad:

# REPLACE this (tinygrad):
#   model_outputs = tinygrad_model.run(img_np)
```

### Trick 3 — CPU Pinned to A76 Big Core (Not Shared With UI/Voice)

**File:** `common/core_config.py`

RK3588 has **4 fast A76 cores** (big) + **4 slow A55 cores** (little). modeld is pinned to the big cores. Voice, UI, navigation etc. are pinned to the little cores. This prevents scheduling jitter.

```python
# common/core_config.py

DAEMON_CORE_TYPE = {
    # ── BIG cores (A76, 2.4 GHz) — Safety-Critical ADAS ──────────────────
    "modeld":    CORE_BIG,   # ← AI inference orchestration on A76
    "controlsd": CORE_BIG,   # ← Vehicle control on A76
    "v4l2d":     CORE_BIG,   # ← Camera capture on A76
    "stereod":   CORE_BIG,   # ← Stereo depth on A76
    "gridd":     CORE_BIG,   # ← Occupancy grid on A76
    "socketd":   CORE_BIG,   # ← CAN bus on A76

    # ── LITTLE cores (A55, 1.8 GHz) — Support Functions ──────────────────
    "waked":     CORE_LITTLE, # Voice wake word — won't delay modeld
    "voiced":    CORE_LITTLE, # STT transcription — won't delay modeld
    "navd":      CORE_LITTLE, # Navigation — won't delay modeld
    "ui":        CORE_LITTLE, # Dashboard UI — won't delay modeld
}
```

**In modeld.py startup:**
```python
# modeld.py — main() function
set_daemon_affinity("modeld")         # ← Pin to A76 big cores
config_realtime_process(DT_MDL, 54)  # ← SCHED_FIFO priority 54 (real-time!)
                                      #   Normal processes = priority 0
                                      #   modeld at 54 = almost never preempted
```

`SCHED_FIFO priority 54` means the OS will NOT interrupt modeld mid-frame to run something else. It runs to completion, then yields. This eliminates latency spikes.

**Orange Pi 5 — add these 2 lines at startup:**
```python
import os
os.sched_setaffinity(0, {4, 5, 6, 7})   # Pin to A76 cores (cores 4-7 on OPi5)
# Or use the same set_daemon_affinity() from common/core_config.py
```

### Trick 4 — DMA-BUF Zero-Copy Camera (No memcpy Before GPU)

**File:** `system/v4l2d/v4l2d.py`

The camera frame goes from sensor → VisionIPC shared memory via DMA-BUF. There is **no CPU copy** at all. The GPU reads directly from the same physical memory the camera wrote to.

```python
# system/v4l2d/v4l2d.py (simplified)

# V4L2 allocates DMA-BUF buffers in kernel (hardware memory)
buf = v4l2_buffer()
buf.type   = V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE
buf.memory = V4L2_MEMORY_DMABUF   # ← DMA-BUF mode: kernel owns the memory

ioctl(fd, VIDIOC_QBUF, buf)   # Queue buffer for camera to fill
ioctl(fd, VIDIOC_DQBUF, buf)  # Wait for camera to fill it (blocking, ~1ms)

# Publish to VisionIPC — this is a pointer, NOT a copy:
vipc_server.send(stream_type, buf.m.planes[0].m.fd, ...)
# ↑ sends the DMA-BUF file descriptor, not the data itself

# In modeld, DrivingModelFrame.prepare() gets:
#   buf.buf.buf_cl   ← cl_mem handle pointing to the SAME physical memory
# The GPU transform reads directly from camera memory. Zero copy!
```

**Without DMA-BUF (what Orange Pi 5 webcam does):**
```
Camera → USB → kernel buffer → userspace copy → numpy array → GPU copy → inference
                               ↑ 3ms            ↑ 5ms          ↑ 3ms
Total extra overhead: ~11ms just in memory copies!
```

**With DMA-BUF (this repo):**
```
Camera → kernel DMA-BUF → GPU OpenCL (same physical memory) → NPU copy → inference
                           ↑ 0ms (no copy!)                    ↑ 1ms
Total overhead: ~1ms
```

**Orange Pi 5 with MIPI CSI camera:**
DMA-BUF is available IF you use a proper MIPI CSI camera (not USB). The v4l2d code works as-is on Orange Pi 5 with a CSI camera.

## 4. How The AI Brain Works — From Camera to Steering

This is the full journey of one camera frame, explained step by step:

```
STEP 1: Camera (hardware)
  OX03C10 sensor captures image (1920x1080, HDR4, 20fps)
  MIPI CSI bus transfers data to chip

STEP 2: v4l2d daemon (system/v4l2d/)
  V4L2 driver reads frame from kernel
  Zero-copy via DMA-BUF (no memory copy!)
  Publishes NV12 frame to VisionIPC shared memory

STEP 3: modeld reads frame (selfdrive/modeld/)
  Gets frame pointer from VisionIPC
  Runs OpenCL GPU transform (warp + YUV->planar)
  Copies preprocessed data to NPU input buffer

STEP 4: NPU runs driving_vision.rknn (NPU Core 0)
  Input: (1, 12, 128, 256) uint8 tensor
  Computes in ~10ms
  Output: 6116 float32 values (features about the scene)

STEP 5: NPU runs driving_policy.rknn (NPU Core 0 or 2)
  Input: vision features + driver actions + traffic conventions
  Computes in ~5ms
  Output: 176 float32 values (how to steer, brake, accelerate)

STEP 6: modeld publishes modelV2 message (cereal msgq)
  Other daemons read this message

STEP 7: controlsd reads modelV2
  Computes desired steering angle and acceleration
  Sends commands to socketd

STEP 8: socketd sends CAN messages to car
  Car's electric power steering receives angle command
  Car steers itself
```

**Simple explanation:** It's like: Camera takes a photo → computer squishes and rotates it → AI brain (NPU) looks at it and decides "turn left 3 degrees" → the software tells the car's steering wheel to turn.

## 6. The NPU — The Dedicated AI Chip

**What is an NPU?** Imagine a calculator that only does one type of math — matrix multiplication — but does it insanely fast. Neural networks are basically huge amounts of matrix multiplication, so an NPU is a perfect match. It's not good for anything else, but for AI inference it's 10x faster than a general CPU.

**RK3588 NPU specs:**
- 3 independent cores
- Each core = 2 TOPS (Tera Operations Per Second)
- Total = **6 TOPS**
- Supports INT8 (8-bit integers, quantized) and FP16 (16-bit floats)
- Format: `.rknn` files (converted from ONNX using rknn-toolkit2)

**Runtime library:** `librknnrt.so`
**Location in repo:** `third_party/rknpu2/runtime/RK3588/Linux/librknn_api/aarch64/librknnrt.so`

**How it's used in code:**
```python
# system/inferenced/rockchip/npu.py
# Tries Python rknnlite API first, falls back to ctypes
from rknnlite.api import RKNNLite
rknn = RKNNLite()
rknn.load_rknn(model_path)
rknn.init_runtime(core_mask=RKNN_NPU_CORE_0)  # Lock to Core 0

# Run inference
rknn.inputs_set([input_array])
rknn.run()
outputs = rknn.outputs_get()
```

**Key RKNN functions:**
- `rknn_init` — Load model, specify which core
- `rknn_inputs_set` — Copy input data to NPU buffer
- `rknn_run` — Execute inference
- `rknn_outputs_get` — Get results back

## 9. The GPU (Mali-G610) — Image Preprocessing Only

**Critical distinction:** The GPU is NOT used for AI inference in this repo. It is used ONLY for image preprocessing.

**What preprocessing means:**
1. **YUV to planar conversion** — Camera gives NV12 (YUV420), AI needs separate Y/U/V planes
2. **Affine warp** — Rotate/scale the image to match what the AI model was trained on
3. **Normalization** — Scale pixel values from 0–255 to what the AI expects

**OpenCL kernel files used:**
```
selfdrive/modeld/
  loadyuv.cl      <- Converts NV12 YUV to planar
  transform.cl    <- Affine transform (warp)
```

**Mali-G610 GPU specs:**
- 10 shader cores
- Supports OpenCL 2.0, OpenGL ES 3.2, Vulkan 1.2
- Shared with the UI (OpenGL for the dashboard display)
- Shared with stereod (OpenCL SGM stereo depth)

**GPU usage by daemon:**
| Daemon | GPU Used For | Time |
|--------|-------------|------|
| `modeld` | YUV+warp preprocessing | ~5 ms/frame |
| `stereod` | SGM stereo depth | ~30 ms/frame |
| `ui` | OpenGL rendering | ~16 ms/frame (60Hz) |

**Important:** modeld and stereod share the GPU. Mali's unified shader design means they time-slice. The Mali scheduler handles this automatically.

## 10. Hardware Abstraction Layer (HAL)

**Files:** `system/hardware/compute.py`, `system/hardware/__init__.py`

The HAL is a **library** (not a daemon). Every AI daemon calls it directly — no message passing needed for compute operations.

**Architecture:**
```
Algorithm Daemons (modeld, stereod, gridd, monod)
    | direct library call (no network, no IPC)
    v
HAL (system/hardware/compute.py)
    | selects the best backend for each operation
    v
Backends:
  RKNNBackend     <- NPU inference (.rknn models)
  ACLCPUBackend   <- ARM NEON optimized CPU (SGM small images, GEMM small matrices)
  ACLGPUBackend   <- Mali OpenCL (SGM large images, GEMM large matrices)
  RGABackend      <- Hardware 2D: resize, crop, format convert (~2ms!)
  MPPBackend      <- Hardware video codec: H.264 encode/decode
  HailoBackend    <- Hailo-8L NPU (RK3576 only)
```

**Backend auto-selection rules:**
| Task | Condition | Backend Chosen |
|------|-----------|---------------|
| Stereo depth (SGM) | Image > 640px | GPU (OpenCL) |
| Stereo depth (SGM) | Image <= 640px | CPU (NEON) |
| Matrix multiply | Matrix > 256x256 | GPU |
| Matrix multiply | Matrix <= 256x256 | CPU |
| Image resize/crop | Always | RGA (dedicated HW, ~2ms) |
| Video encode/decode | Always | MPP (codec HW) |
| Neural net (INT8) | Always | NPU (RKNN) |
| Neural net (FP32) | Always | GPU (OpenCL) |

**Usage in code:**
```python
from openpilot.system.hardware import get_hal, BackendType

hal = get_hal()       # Singleton — initialized once
npu = hal.get_backend(BackendType.NPU)
result = npu.infer('driving_vision', {'input': image})
```

**Previously there was an InferenceD daemon** (see `docs/INFERENCED_ARCHITECTURE.md`). The architecture has since been simplified: the HAL is now a direct library, not a separate daemon process. No msgq overhead for compute.

## 11. Camera System — v4l2d Daemon

**File:** `system/v4l2d/v4l2d.py`
**README:** `system/v4l2d/README.md`

v4l2d is the camera daemon. It talks directly to camera sensors via the Linux V4L2 (Video4Linux2) kernel API.

**What it captures:**

| Camera | Sensor | Resolution | FPS | Color Mode |
|--------|--------|-----------|-----|-----------|
| road | OX03C10 | 1920x1080 | 20 | HDR4 (on-chip) |
| wide_road | OX03C10 | 1920x1080 | 20 | HDR4 (on-chip) |
| stereo_left | GC4653 | 2560x1440 | 20 | SDR only |
| stereo_right | GC4653 | 2560x1440 | 20 | SDR only |
| tele_road | OX03C10 | 1920x1080 | 20 | HDR4 (RK3576 only) |
| side_left | AHD->UVC | — | 20 | SDR (RK3576 only) |
| side_right | AHD->UVC | — | 20 | SDR (RK3576 only) |

**How it finds cameras (dynamic discovery):**
The `/dev/videoN` device number is NOT fixed — it can change on reboot. So v4l2d reads `/sys/class/video4linux/` and looks for devices whose name contains `rkisp_mainpath` AND the sensor name (`ox03c10` or `gc4653`). This is more reliable than hardcoding `/dev/video0`.

**Zero-copy DMA-BUF:**
Frames go from sensor -> kernel -> VisionIPC shared memory without any CPU copying. This saves ~3ms per frame.

**VisionIPC server names:**
- `"v4l2d"` — publishes road, wide_road, tele_road, stereo_left, stereo_right
- `"uvcd"` — publishes driver, side_left, side_right (RK3576 only)

**Who reads from v4l2d:**
| Consumer | What It Reads | Purpose |
|---------|-------------|---------|
| `modeld` | road, wide_road, tele_road | AI driving model |
| `gridd` | stereo_left, stereo_right | Depth + segmentation |
| `stereod` | stereo_left, stereo_right | SGM depth map |
| `monod` | road, wide_road, tele_road | Object detection (Hailo) |
| `driverd` | driver (from uvcd) | Driver monitoring |
| `sided` | side_left/right (from uvcd) | Blind spot detection |

## 12. Camera Sensors Detail (OX03C10 and GC4653)

### OX03C10 (Mono cameras: road, wide_road, tele_road)

**File:** `system/v4l2d/drivers/ox03c10.py`

- Omnivision OX03C10 CMOS sensor
- **On-chip HDR4** — does 4-exposure high dynamic range INSIDE the sensor chip
- Result: 140dB dynamic range (can see in bright sun AND deep shadow simultaneously)
- The ISP (RKISP) sees a LINEAR image — it does NOT do HDR processing
- Control: V4L2 private control `V4L2_CID_PRIVATE_BASE + 1` sets HDR mode

**HDR modes:**
```python
class HDRMode(Enum):
    LINEAR = 0   # Normal SDR (no HDR)
    HDR2 = 1     # 2-exposure blend
    HDR3 = 2     # 3-exposure blend
    HDR4 = 3     # 4-exposure blend (DEFAULT — maximum quality)
```

**Why HDR4?** Driving involves extremely varied lighting. Tunnels (dark), sun glare (bright), oncoming headlights at night. Normal cameras would be either overexposed or underexposed. HDR4 captures all of it at 140dB.

### GC4653 (Stereo cameras: stereo_left, stereo_right)

- Galaxycore GC4653 CMOS sensor
- **SDR ONLY** — no HDR capability
- Why no HDR for stereo? Because HDR uses multiple exposures at different times. If left and right cameras have slightly different timing, the depth calculation breaks. At 100 km/h relative speed, a 10–15ms timing error = **+/- 0.8m depth error** (5x worse). Safety risk!
- The two stereo cameras are synchronized via I2C to ensure they capture at exactly the same moment

## 13. ISP and HDR — Why Mono and Stereo Cameras Work Differently

**File:** `docs/eop/03_Software/Architecture/CAMERA_ISP_HDR_ARCHITECTURE.md`

**ISP = Image Signal Processor** (the RKISP chip inside RK3588)

The pipeline has two separate paths:

```
Mono cameras (road, wide_road, tele_road):
  OX03C10 sensor -> HDR4 blend (on-chip) -> RKISP (linear mode) -> NV12 frame

Stereo cameras (stereo_left, stereo_right):
  GC4653 sensor -> SDR direct -> V4L2 direct -> NV12 frame (no ISP processing)
```

**RKIAQ ISP 3A (Auto-Exposure, Auto-White-Balance):**
- Uses `librkaiq.so` (Rockchip ISP library)
- **Status: STUBBED** — `ISP_AVAILABLE = False` in current code
- When implemented: hardware-level brightness and color auto-correction
- Without it: camera image quality may be inconsistent in changing light

**Critical safety rule:**
> Never switch HDR mode while the car is engaged (driving). HDR mode switching takes ~500ms and drops frames. `selfdrived` blocks HDR changes when `controlsState.enabled = True`.

## 16. Stereo Depth — stereod

**File:** `selfdrive/stereod/stereod.py`, `stereod/sgm.py`
**README:** `selfdrive/stereod/README.md`

### What Is Stereo Depth?

Imagine holding two fingers in front of your face and closing each eye alternately. The closer the finger, the more it shifts between your eyes. Two cameras do the same thing — by comparing what the left and right cameras see, you can calculate how far away each object is.

### Algorithm: SGM (Semi-Global Matching)

SGM is a classic computer vision algorithm that finds the best match between left and right camera images:

1. **Census Transform** — Each pixel is described by comparing it to its 5x5 neighbors
2. **Cost Volume** — Calculate how different each pixel is between left and right (using Hamming distance)
3. **Cost Aggregation** — Smooth the matches (4-path or 8-path aggregation)
4. **Winner-Takes-All** — Pick the best match = depth estimate
5. **Post-processing** — 3x3 median filter to remove noise

### Hardware Used

| Platform | GPU | Baseline | Resolution | Latency |
|---------|-----|---------|-----------|---------|
| RK3588 | Mali-G610 | 80 mm | 640x480 | **~30 ms** |
| RK3576 | Mali-G52 | 160 mm | 640x480 | **~35 ms** |

The wider baseline (160mm vs 80mm) on ExoPilot 02 gives better depth accuracy at long range.

### Outputs

- `stereoDepth` — disparity map (float32), confidence map, at 20 Hz
- `stereoDetections` — 2D YOLO object boxes
- `stereoSegments` — PP-LiteSeg segmentation masks (19 classes)

### Fault Safety

**If GPU fails -> IMMEDIATE_DISABLE** (3 consecutive failures = stop)
There is NO CPU fallback in production (SGM on CPU = ~500ms = too slow for safety).

## 17. Occupancy Grid — gridd

**File:** `selfdrive/gridd/gridd.py`
**README:** `selfdrive/gridd/README.md`

### What Is An Occupancy Grid?

Imagine looking at your car from above (bird's eye view). The occupancy grid is a 2D map of what's around the car. Each cell in the grid says "is there something here? Yes/No (and how confident?)".

It's like a radar screen — except it uses cameras and math instead of radio waves.

### How gridd Works

```
stereo_left + stereo_right (from v4l2d)
    v
Lazy BEV reprojection (lazy_bev.py)
    "Only compute the 3D positions we actually need"
    v
Probabilistic Bayes filter (temporal consistency)
    "Remember what we saw before, don't freak out over one bad frame"
    v
Optional fusion with monod (if Hailo available)
    v
gridObjects message -> pathd, controlsd
```

**Key design choice — "Lazy" BEV:**
Normal approach: reproject EVERY pixel from 2D to 3D (expensive!)
Lazy approach: only reproject the pixels needed for the occupancy grid (efficient)

### Why This Matters for Safety

`gridObjects` is a **CRITICAL PATH** message. ADAS (automatic emergency braking, etc.) depends on it. If gridd crashes, the system falls back gracefully rather than making unsafe decisions.

## 19. Driver Monitoring — driverd

## 22. Navigation — navd / mapd / pathd / globald

### navd (Turn-by-Turn Navigation)

**Valhalla** offline routing engine + OpenStreetMap maps
**Voice announcements** at 3 distances:
- ~500m: "In 500 meters, turn right onto Main Street"
- ~200m: "In 200 meters, turn right"
- ~80m: "Turn right"
- <50m: "You have arrived"

Published as `ttsRequest` with priority=1 (high), interrupt=True

### mapd (Map Tile Management)

Downloads and caches OSM tiles for the current location. Also provides:
- Speed limit data from OSM tags
- Road type (highway, city street, etc.)

**Speed limit stub:** `mapd.py` line 253 — next speed limit always returns `0` (not yet implemented)

### pathd (Complex Path Planning)

Uses **Hybrid A*** algorithm for complex maneuvers. Reads from:

- `gpsLocation` (from pigeond)
- OSM road geometry (from mapd/globald)

`groundd` daemon (road edges, lane lines) — not implemented. `pathd` handles `None` gracefully.

### globald (Global Position Fusion)

Fuses GPS + OSM for precise global position. Works with `rtkd` for RTK centimeter-precision GPS (when RTK correction data available).

## 27. Safety Architecture (Two-Layer)

This is NOT standard openpilot's single Panda safety layer.

| Layer | Component | Limits | Role |
|-------|-----------|--------|------|
| **Layer 1 (Software)** | `socketd` Python daemon | 80% of Panda limits (TIGHTER) | Primary control: catches software bugs |
| **Layer 2 (Hardware)** | TC275 Gateway chip | 100% of Panda limits (LOOSER) | Hardware safety net |

**How it works:**
- socketd (software) enforces strict limits. If it sends a command that's too aggressive, TC275 (hardware) will reject it.
- TC275 is a dedicated automotive safety microcontroller. It cannot be reprogrammed by software bugs.
- Even if socketd goes completely wrong, TC275 caps the output.

**CAN bus:** All vehicle communication goes through `socketd` -> CAN bus -> car's ECU.

## 30. How To Make Your Orange Pi 5 Work Like This

Your Orange Pi 5 has an **RK3588S** chip — a cousin of the RK3588 used here. You CAN use many of these ideas. Here's exactly what to do:

### What The Orange Pi 5 Has

| Feature | Status |
|---------|--------|
| RK3588S NPU (6 TOPS) | YES — Available! Same RKNN2 runtime |
| Mali-G610 GPU | YES — Available! Same as ExoPilot 01 |
| RGA3 hardware accelerator | YES — Available! Same as ExoPilot 01 |
| MPP video codec | YES — Available! |
| MIPI CSI camera ports | YES — 2–4 depending on board revision |
| On-chip HDR sensors | DEPENDS — need OX03C10 module |
| Hailo-8L | NO — not included (but can add via PCIe M.2) |

### Step 1: Switch From tinygrad/OpenCL to RKNN NPU

**Current Orange Pi 5 approach (slow):**
```python
# sunnypilot-pc: tinygrad runs model on Mali GPU via OpenCL
# GPU=1 ./launch_openpilot.sh  <- Uses OpenCL, ~50-150ms
```

**New approach (10x faster):**
```bash
# Install rknnlite on Orange Pi 5 (Ubuntu)
pip install rknn-toolkit-lite2  # or rknnlite from Rockchip
```

```python
# Replace tinygrad inference with RKNN
from rknnlite.api import RKNNLite

rknn = RKNNLite()
rknn.load_rknn('driving_vision.rknn')  # Use this repo's model!
rknn.init_runtime(core_mask=RKNNLite.NPU_CORE_0)

# Run
rknn.inputs_set([image])
rknn.run()
output = rknn.outputs_get()
```

**You need:** The `.rknn` model files from `models/rknn/` in this repo.
The RK3588S on Orange Pi 5 uses the SAME librknnrt.so as ExoPilot 01.

### Step 2: Use RGA for Image Preprocessing (Instead of OpenCV)

**Current:** OpenCV resize/crop (10–12 ms on CPU)
**Better:** RGA hardware (2 ms!)

```bash
# Install RGA Python bindings
pip install pyrga  # or build from Rockchip's librga source
```

```python
import pyrga

# Hardware resize: 1920x1080 -> 640x480 in 2ms (was 10ms)
output = pyrga.resize(frame, 640, 480)

# Hardware format convert: NV12 -> RGB in 2ms
output = pyrga.convert(nv12_frame, 'nv12', 'rgb')
```

### Step 3: Replace USB Webcam With MIPI CSI Camera

**Current:** USB webcam (30ms USB overhead, SDR only)
**Better:** MIPI CSI camera (1ms, optional HDR)

**Compatible cameras for Orange Pi 5:**
- OV5647 (basic, cheap, SDR)
- IMX219 (better quality, SDR)
- OX03C10 (same as this repo, HDR4! — needs adapter)

To use v4l2d dynamic discovery on Orange Pi 5:
1. Copy `system/v4l2d/v4l2d.py` from this repo
2. It reads `/sys/class/video4linux/` — works on any Linux
3. Set sensor name filter to match your sensor

### Step 4: Use SGM Stereo Depth (If You Add a Second Camera)

If you add a second MIPI CSI camera (stereo pair):

```python
# Copy from this repo: selfdrive/stereod/sgm.py
from openpilot.selfdrive.stereod.sgm import SGM, SGMConfig

config = SGMConfig(target_width=640, target_height=480, max_disparity=64)
with SGM(config, target="auto") as sgm:
    # "auto" picks GPU if Mali OpenCL available, else CPU
    result = sgm.compute(left_frame, right_frame)
    depth_map = result.disparity  # ~30ms on Mali-G610
```

### Step 5: Platform Detection — Make Orange Pi 5 Recognized

In `system/hardware/registry.py`, add Orange Pi 5:
```python
# /proc/device-tree/compatible on Orange Pi 5 contains "rockchip,rk3588s"
# Add to the detection map:
if 'rk3588s' in compatible:
    return RK3588Hardware()   # RK3588S is software-compatible with RK3588
```

Or simply set the environment variable:
```bash
export HARDWARE=rk3588
python system/manager/manager.py
```

### Step 6: What You CANNOT Do on Orange Pi 5 (Without Extra Hardware)

| Feature | Why Not Available |
|---------|-----------------|
| OX03C10 HDR camera | Needs specific MIPI adapter + kernel driver |
| Hailo-8L NPU | Separate PCIe card (~$100) + driver install |
| TC275 safety gateway | Custom hardware in ExoPilot |
| RTK GPS | Separate module needed |
| 4-camera array | Need camera expansion board |

### Summary: Orange Pi 5 Upgrade Path

```
Current (slow, ~100-200ms):
  USB webcam -> OpenCL tinygrad -> 1 model -> steering

Better (fast, ~20-30ms):
  MIPI CSI camera -> DMA-BUF -> RKNN NPU -> vision + policy -> steering

Best (full EOP features):
  2x MIPI CSI (stereo) -> DMA-BUF -> RKNN NPU (vision+policy)
                                   -> Mali GPU (SGM depth)
                                   -> RGA (fast resize)
                                   -> CPU (occupancy grid)
                        -> steering + depth-aware obstacle avoidance
```

**The biggest single win:** Replace tinygrad OpenCL inference with RKNN NPU inference. Same model, 5–10x faster. This repo's `models/rknn/driving_vision.rknn` and `driving_policy.rknn` work directly on Orange Pi 5's RK3588S NPU.
